<h1 class="gw-heading-l">Button</h1>

Use buttons for actions. Use links for navigation. There is one primary action per decision area.

<ButtonHeroDemo />

<h2 class="gw-heading-m">When to use Button</h2>

Use a button when the user is committing to an action — saving, submitting, deleting, confirming. Write the button text so it names the action: `Save and continue`, `Delete account`, `Send claim`. Avoid generic labels like `Submit` or `OK`.

Use exactly one primary button per decision area. A decision area is a form, a dialog, or a card asking the user to do something — not a whole page. Multiple decision areas on a page can each have their own primary button.

<h2 class="gw-heading-m">When not to use Button</h2>

Do not use a button to navigate to another page. Navigation belongs in links — `<a>` elements with `href`. A button is a commitment; a link is a destination. Confusing the two breaks browser conventions (right-click, middle-click, copy-link-address) and accessibility expectations.

Do not stack multiple primary buttons in the same decision area. If two actions feel equally important, one of them is probably a secondary action — or the page is asking too many things at once. Split it.

Do not reach for `variant="destructive"` for "save" actions you suspect users will regret. Destructive styling is for irreversible actions like deletion. Misusing it dulls the signal for actions that genuinely need it.

<h2 class="gw-heading-m">How it works</h2>

The variant set is closed: `primary`, `secondary`, `destructive`, `ghost`. Pick by visual weight and intent, not by colour. Variants are visual treatments — the underlying element is always `<button>`. For navigation, use a link, not a button styled as one.

<h3 class="gw-heading-s">Primary</h3>

The default. Use it for the action that completes what the user came here to do — saving a form, sending a message, confirming a choice.

`primary` is the default value of the `variant` prop; you do not pass it explicitly.

<ButtonPrimaryDemo />

<h3 class="gw-heading-s">Secondary</h3>

A supporting action that sits alongside a primary, like `Save as draft` next to `Save and continue`. Use sparingly: too many secondary buttons flatten the hierarchy and make the primary action hard to find.

If a secondary action would only ever be used by a small fraction of users, consider a ghost button or a link instead — keep the visual weight low.

<ButtonSecondaryDemo />

<h3 class="gw-heading-s">Destructive</h3>

For irreversible actions — deleting an account, discarding draft work, cancelling a subscription. Pair with a confirmation step so the action is never a single click.

Always provide an escape. Pair a destructive button with a ghost `Cancel` so the user can back out without committing.

<ButtonDestructiveDemo />

<h3 class="gw-heading-s">Ghost</h3>

A tertiary action with low visual weight. Use for `Cancel`, `Back`, `Close`, or alternative paths that do not commit. Place it next to a primary so the visual hierarchy stays clear.

A page or dialog with only ghost buttons has no primary action. That is usually a sign the page is asking the user to navigate, not act — use links instead.

<ButtonGhostDemo />

<h3 class="gw-heading-s">Grouping buttons</h3>

When two or more buttons sit together, wrap them in a `<div class="gw-button-group">`. The wrapper lays them out horizontally with consistent spacing and wraps individual buttons to the next line on narrow viewports rather than overflowing.

A plain link (`<a class="gw-link">`) inside the group aligns with adjacent buttons, so a `Cancel` link reads as part of the same decision area as the action button it sits next to — useful when the cancel path is genuinely a navigation away from the current screen, not a no-op cancel of the in-page action.

<ButtonGroupDemo />

<h3 class="gw-heading-s">Disabled state</h3>

The `disabled` HTML attribute applies to every variant. Use it only when the reason a button is disabled is visible elsewhere on the page — for example, a form invalid because of an unresolved error already shown above.

If the reason is not obvious, leave the button enabled and explain the constraint in error copy after the user activates it. Disabled buttons hide reasons; visible errors do not.

<ButtonDisabledDemo />

<h3 class="gw-heading-s">Prevent accidental double submission</h3>

Users can submit twice for three different reasons, each of which needs a different fix:

- **Fast double-clicks** within a few hundred milliseconds — motor impairments such as hand tremors, or habituated double-clicks from desktop operating systems. Pass `preventDoubleClick` to ignore clicks that fire within 500ms of the previous accepted click. The demo below sends one invitation per slow click, no matter how fast you double-click.
- **Slow-connection resubmits** seconds later, where the user clicks again because they got no feedback. `preventDoubleClick` does not catch this case and should not — the user genuinely intends the second click. Manage a loading state in the consumer and pass `disabled` while the request is in flight.
- **Server-side dedupe** is needed regardless of the client. Use an idempotency key or equivalent so a duplicate request cannot commit a duplicate transaction even if it reaches the server.

<h3 class="gw-heading-s">Accessibility</h3>

Focus is a 3px solid yellow ring. It is loud on purpose and identical across every interactive element in Groundwork — that consistency is what makes it unmissable.

Write button text that describes the action specifically. `Save address` beats `Submit`; `Delete account` beats `Continue`. Screen-reader users navigating by buttons hear the text out of context, so the text alone must answer "what happens if I press this?".

Destructive variants do not rely on colour alone. The text must make the action's seriousness clear, because users who cannot see the red treatment — colour-blind users, users on monochrome devices, users with high-contrast OS themes — only have the words.
