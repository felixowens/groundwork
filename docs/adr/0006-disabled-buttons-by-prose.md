# Disabled buttons are governed by prose, not by API

Groundwork strips `className` and `style` from component props because the failure modes are common, easy to type-check, and have no legitimate use case. The `disabled` HTML attribute is left intact on Button and governed by prose only — the Button page says "use it only when the reason a button is disabled is visible elsewhere on the page" and the type system does nothing to enforce that. This is the only editorial rule on the Button page without structural backing.

A `disabledReason: string` prop required when `disabled={true}` was considered. Two render strategies were rejected: rendering the reason visibly would push Groundwork into deciding the layout of reason copy, conflicting with form-level error blocks the consumer already manages; wiring only `aria-describedby` would add API ceremony without enforcing visibility for sighted users. Neither cleanly handles transient disabled states like a button disabled while a save is in flight, where the "reason" is an in-flight state rather than a contextual constraint.

The asymmetry with `className`/`style` is real but justified: those have no use case Groundwork accepts; `disabled` does, and the rule about when to use it is contextual rather than absolute. Re-evaluate if a `loading` or pending-state prop arrives that absorbs the transient-disabled case (potentially leaving only the persistent-disabled cases that benefit from structural enforcement), or if a consumer pattern surfaces where the ceremony of `disabledReason` would clean up more than it costs.
