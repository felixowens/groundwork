# Brands are primitive-token deltas, not component variants

INTENT.md bets that re-theming Groundwork for a different visual identity is "changing a handful of primitive values, not auditing every component." This is the decision that tests that bet in the open: a **brand** is a single `tokens/themes/<name>.json` delta that overrides private primitive ramps (and, rarely, a few semantic mappings). The build emits the default theme on `:root` and every brand under `[data-gw-theme="<name>"]`; because components read semantic tokens and semantic tokens resolve their `var(--_gw-*)` references at the point of use, overriding the privates under a scope re-skins everything below it with no component CSS touched. A `Brand` control in the docs masthead sets the attribute on `<html>`; light and dark continue to follow `prefers-color-scheme` per brand. Six brands ship as a switchable gallery so the claim is dogfooded, not asserted.

The test paid for itself in findings. Renaming the entire primitive layer from appearance-names to role-names (`blue→accent`, `red→danger`, `green→positive`, `amber→caution`) — a prerequisite so brands share one constant semantic mapping — touched **two lines of doc prose and zero components**, and the default palette came out byte-identical. That is the components-reference-semantic-tokens-only rule paying out exactly as designed, and it also retired a latent violation of the names-describe-decisions principle. One genuinely missing primitive surfaced: brands need a heading typeface, so a `--font-display` semantic token was added (defaulting to the sans stack, leaving the default theme unchanged). It is a global semantic token, not a component token, so it stays within the two-tier rule.

The sharp finding is that **free re-theming does not preserve the accessibility contract**. `--action` does double duty — the fill behind white button text (wants a mid-tone) and the colour of link text on a light surface (wants something dark for 4.5:1) — so a vivid brand accent silently drops link contrast below WCAG AA. Every brand accent here was therefore tuned to a measured ≥4.5:1 in both roles (verified, not eyeballed; some prototype hues — orange, yellow, mint — had to darken or shift to clear it). A theme-contrast check belongs in CI as a guardrail, and if a future consumer wants vivid fills *and* legible links, that is the trigger to split `--action` into a fill token and an AA-safe text token rather than overloading one.

The boundary the exercise drew is the durable result: the token system re-skins **colour, shape, type scale, elevation, and border weight** for free, but **treatment-level signatures are not tokens**. The prototype brands' mono/uppercase buttons, tab-styled and slash-prefixed eyebrows, ink section-bands, and button drop-shadows were dropped from the shipped brands. Those are either page composition (the consumer's job) or would require an explicit, approved set of treatment tokens — which the "no component tokens without approval" rule correctly gates. So Groundwork supports re-*colouring*, re-*shaping*, and re-*typesetting* a brand, not re-*structuring* one, and that is the right line to hold for a constraint-driven system.

Two scope choices follow from keeping light/dark on `prefers-color-scheme` (a manual scheme toggle was declined): brands are scheme-*agnostic palettes*, so a brand cannot force dark — "Midnight" is a cool, mint-accented palette that is luminous under a dark system preference and calm under a light one, not a dark-locked theme. And the brand webfonts load from an external stylesheet in the docs `<head>`, which is why the `/themes/` gallery is excluded from visual snapshots (its rendering is environment-dependent online vs offline); brand rendering is verified manually. Revisit if a real project needs a force-dark brand (re-open the scheme toggle), when promoting the contrast check to CI, or to self-host the brand fonts and remove the external dependency.
