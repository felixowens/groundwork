Design System v0.2

Groundwork

A foundation library for web applications. Two-tier token architecture, accessible components, and patterns that make the right thing easy and the wrong thing hard.

Token architecture

Two layers. Primitives hold raw values. Semantic tokens hold decisions. Components only reference the semantic layer. Dark mode remaps which primitives the semantic tokens point to — components don't change.

┌───────────────────────────────────────────────────────┐ │ PRIMITIVES (private — prefixed --_gw-) │ │ │ │ Palette: _gw-neutral-0 … _gw-neutral-950 │ │ Brand: _gw-blue-100 … _gw-blue-600 │ │ Status: _gw-red-*, _gw-green-*, _gw-amber-* │ │ Scale: _gw-space-1 … _gw-space-20 │ │ Type: _gw-text-xs … _gw-text-3xl │ │ Borders: _gw-border-thin … _gw-border-heavy │ │ │ │ ⚠ Never reference in component CSS. │ ├───────────────────────────────────────────────────────┤ │ SEMANTIC TOKENS (public) │ │ │ │ Ink: --ink, --ink-secondary, --ink-disabled │ │ Surface: --surface, --surface-raised, │ │ --surface-sunken, --surface-overlay │ │ Border: --border, --border-strong │ │ --border-width-thin … heavy │ │ Action: --action, --destructive, --success … │ │ Focus: --focus-ring-color, width, offset │ │ Type: --text-body, --text-heading … │ │ │ │ ✓ Components reference ONLY these. │ ├───────────────────────────────────────────────────────┤ │ DARK MODE │ │ │ │ Remaps semantic → different primitives. │ │ --ink: _gw-neutral-800 → _gw-neutral-50 │ │ --surface: _gw-neutral-0 → _gw-neutral-900 │ │ Components don't change. │ └───────────────────────────────────────────────────────┘

Principles

1
Constraints over choices

A closed token set. If it's not in the scale, it doesn't exist.

2
Two layers, one seam

Primitives hold values. Semantic tokens hold decisions. Theming swaps at the seam.

3
Accessibility as architecture

Components look broken without labels. Focus is unmistakable. Colour is never the only signal.

4
Names describe decisions

--ink-secondary not --ink-muted. --text-body not --text-16. Rename the value without renaming the token.

5
Documentation is product

Every component: when to use, when not to, the reasoning, edge cases.

Components

Button

When to use: For actions. One primary per view. When not to use: Navigation — use links. Rule: Text describes the action ("Save address", not "Submit").

Form fields

Pattern: Every input in .gw-field with a label. Width hints signal expected input length. A bare input looks wrong.

Accordion

What counts as a breaking change?

Removing a token, renaming a class, or altering component markup. Visual tweaks within tokens are not breaking.

Can I add custom tokens?

Yes, in a separate namespace (--app-*). Core tokens are locked.

How do I re-theme?

Override the primitives. Replace --_gw-blue-* with your brand. Semantic tokens inherit automatically.

Tabs

ARIA roles for screen readers. JS toggles panels as enhancement.

Using Groundwork

Getting started

One CSS file. No build step. No dependencies.

<link rel="stylesheet" href="groundwork.css"> <div class="gw-width"> <main class="gw-stack--lg"> <div class="gw-field"> <label class="gw-label" for="email">Email address</label> <span class="gw-hint">We'll send your receipt here</span> <input class="gw-input gw-input--w30" id="email" type="email"> </div> <button class="gw-button">Continue</button> </main> </div>

Re-theming

Override primitives. Everything inherits.

/* your-brand.css */ :root { --_gw-blue-500: #7c3aed; /* violet */ --_gw-blue-600: #5b21b6; --_gw-blue-300: #a78bfa; --_gw-blue-200: #c4b5fd; } /* Done. Buttons, links, tabs, focus — all update. */

The rules

Only reference semantic tokens

No raw hex. No var(--_gw-*) in components. Semantic tokens only.

Every input needs a label

Wrap in .gw-field with .gw-label. Orphaned inputs look broken.

Error messages explain the fix

"Enter your email address" not "Invalid input".

Colour is never the only signal

Text, icons, or structural cues always accompany colour.

One primary action per view

Two primary buttons means one is wrong.
